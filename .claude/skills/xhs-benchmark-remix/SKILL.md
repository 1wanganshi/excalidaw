# 小红书对标采集与二创 Skill

这个技能负责把“小红书关键词找对标 → 采集整理 → 结构分析 → 文案与图片二创 → 交付发布包”变成稳定流程。

目标不是把每次请求都做成重型项目，而是先识别用户属于哪一种任务，再只展开必要层级：

- **咨询模式**：用户想分析对标方向、判断值不值得采、提炼爆款结构、改写标题正文、设计二创策略。
- **执行模式**：用户明确要跑采集、下载素材、批量二创、整理表格、输出文件夹或发布包。
- **排障模式**：用户遇到 xhs-cli、Agent-Reach、cookies、风控、搜索无数据、GPTimage2 接口等问题。

## 重要边界

- 只采集用户有权限访问的公开内容或用户已登录账号能正常查看的内容。
- 遵守平台规则、robots/服务条款和当地法律；不要绕过付费墙、验证码、风控或访问控制。
- 二次创作应做结构借鉴、主题重写、视觉再设计，避免逐字搬运、洗稿、商标/肖像侵权或冒充原作者。
- 输出里保留来源链接、对标原因和风险说明，方便用户人工复核与合规判断。
- 若涉及直接发布，提醒用户检查版权、肖像权、品牌词、医疗/金融/功效承诺等敏感风险。
- 不要打印 cookie 值，不要把 API Key 写进交付文件、日志或技能文件。用户提供的 FMGO / GPTimage2 Key 只能临时用于本次接口调用，后续写文档时只记录变量名和用法，不记录密钥明文。

## 触发后的第一步：先分流，不要直接拉满全流程

先判断用户到底要什么，再进入对应模式。

### A. 咨询模式
适用于：
- “帮我看看这个赛道怎么找小红书对标”
- “分析这篇爆款为什么能火”
- “按这个爆款结构改写 3 个版本”
- “这组素材适不适合做 GPTimage2 二创”

在这个模式里：
- 不默认做真实采集。
- 不默认检查登录、cookies、风控。
- 不默认创建完整项目目录或发布包。
- 只交付用户此刻需要的分析、改写、提示词、结构拆解或轻量表格。

### B. 执行模式
适用于：
- 用户明确要求批量按关键词采集小红书图文/高赞高藏笔记。
- 用户要求下载整理素材、做批量文案二创、图片二创。
- 用户要求输出 CSV/XLSX、文件夹、发布包、矩阵内容素材。
- 用户已经说“直接跑一版”“按这个流程给我产出交付物”。

在这个模式里：
- 先补齐最少必要信息；若用户不想多聊，用默认轻量版起跑。
- 根据是否要真实采集、是否有参考图、是否要批量出图，决定是否进入更重的子流程。
- 需要真实采集时，才检查登录与风控。
- 需要批量出图时，才读取 GPTimage2 相关说明与脚本。
- 交付结果按发布包结构组织，便于后续继续迭代。

### C. 排障模式
适用于：
- “打不开了”“搜索不到”“xhs status 报错”
- “Agent-Reach 装了但抓不到”
- “cookie 明明有但还是 not_authenticated”
- “GPTimage2 接口慢/失败/没落盘”

在这个模式里：
- 先判断问题属于登录、风控、工具安装、接口异常还是任务规模过大。
- 需要时再阅读 `references/runbook.md`。
- 只做与当前故障相关的检查，不要顺手把整条生产流程都跑一遍。

## 先问清楚的最少信息

如果信息不全，最多问 3 个问题，不要把流程卡死：

1. **关键词/赛道**：例如“收纳好物、AI 绘画副业、装修避坑”。
2. **目标交付**：只要分析建议，还是要 CSV / 文件夹 / 发布包？
3. **二创方向**：保留主题换表达，还是换人群 / 换场景 / 换卖点？

可选信息：账号人设、目标人群、图片风格、输出比例、是否需要标题库、是否需要发布表格、是否必须真实抓取小红书数据。

## 默认执行策略：轻量起跑，再按需加深

如果用户说“先跑一版”“别问太多”，不要一上来就跑满重流程。默认按**轻量执行版**处理：

- 每个关键词先采 5 条图文，而不是默认 20 条。
- 先筛 3 条做结构分析，而不是默认 10 条。
- 先产出 1-2 条二创内容，而不是默认 5 条。
- **配图不是可选项**：只要进入小红书二创/发布包执行模式，每篇都必须有实际图片或明确待补图方案。用户已指定 FMGO `gpt-image-2` 为默认真实出图路线时，必须调用 FMGO 图片接口生成实际 PNG/JPG，并校验落盘；不得用 Python/PIL/canvas 在图片上排版写字来冒充生图。没有可用接口或接口失败时，配图提示词、脚本、manifest、补图清单只能放入 `_internal` / `_work` 工作区，不进入最终交付包。
- 默认输出轻量版 CSV 或整理结果；只有用户明确要整包交付时，再展开完整工作区。最终发布包必须清理为“图片 + 发布包”两类文件，不把过程文件交给用户当最终成果。

如果用户确认要全量执行，再扩展到批量采集、批量图片二创、完整发布包。

## 推荐工具路线

优先用用户本地已具备的能力，不重复造轮子。

### 采集工具优先级
1. **OneAPI / GetOneAPI（当前主路径）**：当用户要稳定、可付费、低成本的小红书真实采集时，优先使用 OneAPI。它已实测可用：搜索 `/api/xiaohongshu/search_note` 返回 20 条；详情 `/api/xiaohongshu/fetch_video_detail` 成功返回标题、正文、作者、点赞、收藏、评论、分享、图片等字段；搜索/详情/评论普通接口价格约 0.08 元/次。执行时用 `Authorization: Bearer <API_KEY>`，不要在回复、技能、日志、交付文件里写密钥。
2. **Just One API（备用路径）**：已实测搜索可用，但搜索约 0.2 元/次，详情/评论容易触发 `TOO FAST`，适合作为 OneAPI 不可用时的备用。
3. **本地 xhs-cli / MCP / Playwright（兜底验证）**：适合低成本人工复核、链接读取和发布，但登录态/cookie/风控会影响稳定性。
4. **Apify / 企业数据服务（批量或企业级备用）**：适合大批量、舆情监测、竞品长期监控。
5. **公开资料兜底**：无法真实抓取时使用，但必须诚实标注不是直接抓到的小红书高赞数据。

### 已知工具与故障资料
- 登录、风控、Agent-Reach 安装、公开资料兜底、GPTimage2 批量脚本：读 `references/runbook.md`
- 工具路线、商业 API 稳定性评估、小样本验收标准：读 `references/tool-research.md`
- 任务简报模板：读 `templates/task_brief_template.md`
- 完整执行/交付规范、热点编排流程：读 `references/delivery-workflow.md`

## 执行模式工作流

只在用户明确需要执行时展开，且按需读取对应参考文件。

### 1. 建立任务简报
先用 `templates/task_brief_template.md` 填好：关键词、交付目标、账号定位、二创方向、执行状态。

### 2. 判断真实采集还是兜底模式
- 如果用户明确要求真实小红书数据，优先走 OneAPI 主路径，除非用户指定其他工具或 OneAPI 不可用。
- OneAPI 调用原则：先搜索，缓存结果，再筛选；只对 Top 笔记抓详情；只对少量高价值笔记抓评论。每次调用前估算成本，避免无脑批量扣费。
- 如果登录失效、账号风控、接口异常或用户不要求真实数据，可切到公开资料兜底模式，但必须在交付中标明来源与限制。
- 登录与风控检查细节放在 `references/runbook.md`，不要每次默认展开。

### 3. 采集、热点补充与筛选
- 收集标题、正文、作者、链接、发布时间、互动数据、图片链接/本地路径、标签等。
- 做排序、去重、对标理由说明。
- 如果用户要求“当日热点/热点编排/蹭热点/结合热点”，或内容主题明显适合借势，应增加热点补充步骤：先找当日与赛道相关的热点、热词、事件、讨论情绪，再判断能否自然接入，不要硬蹭。
- 热点只作为切入角度和传播钩子，最终内容必须落到可收藏的判断、方法、清单、步骤或避坑原则上。
- 如果任务只是咨询或轻量试跑，不必默认把所有字段都采满。
- 具体字段、评分、目录建议见 `references/delivery-workflow.md`。

### 4. 爆款结构分析
- 提炼标题钩子、首图策略、正文结构、评论触发点、可迁移与不可照搬元素。
- 输出给后续文案二创和图片二创使用。

### 5. 文字二创
- 保留爆款语气、内容核心和抓人节奏，但换表达、换细节、换账号视角。
- 至少产出标题候选、正文、标签、评论区引导。
- 避免逐句洗稿，先抽象内容骨架再重写。

### 6. 配图与 GPTimage2 二创（必做环节）
- **配图是发布包的必交付物，不是可选增强项**。只要用户要小红书二创、图文、发布包、整包交付、可直接发布素材，每篇内容都必须有配图成果。
- **用户指定的真实出图默认路线**：当用户提供或指定 FMGO 图片接口时，默认使用 `https://www.fmgo.top/v1` + `gpt-image-2` 生成图片。API Key 只能通过环境变量、临时命令或安全配置读取，不得写入技能文件、交付文件、日志说明或最终回复。
- **禁止 Python 伪出图**：不得用 Python / PIL / canvas / HTML 截图等方式把文字手工画到图片上来冒充 AI 生图。Python 脚本最多只能做 API 调用、保存返回图片、校验文件、整理 manifest；图片上的所有正文、标题、排版文字都必须写进作图提示词，由 `gpt-image-2` 在生成图片时完成。
- 配图成果分两级：
  1. **最终实际图片**：如果存在 FMGO API Key、`GEEKNOW_API_KEY` / `OPENAI_API_KEY` / 其他可用出图工具，应调用出图流程生成 PNG/JPG，复制到最终交付的 `图片/` 目录，并检查文件真实落盘、数量足够、可打开。用户指定 FMGO 时，优先 FMGO `gpt-image-2`。
  2. **内部待补图资料**：只有在没有 API Key、接口失败、真实采集失败或暂时不能出图时，才允许降级；`图片内容脚本.md`、`配图提示词.json`、`补图清单.md`、manifest 等只能放入 `_internal` / `_work` 工作区。最终交付仍应只暴露 `图片/`（或缺图占位说明）和 `发布包.txt`，不能把中间文件混进最终发布包。
- 先识别原图抓眼球点和画面结构，再写 image2image 提示词；没有真实参考图时，按原创多图卡片生成/规划，并标注“原创生成/待 image2image 补跑”，不要冒充真实图生图。
- **图文笔记默认不是单图交付**：如果对标样本是多图笔记，或用户说“图文/整篇/发布包”，应按多图笔记处理；每篇发布内容独立生成一组风格一致的图片，而不是只做一张封面。
- **图片即正文**：小红书图文里，图片承担主要阅读任务。多图应按“强钩子图 → 症状/判断图 → 方法/清单图 → 行动/总结图 → 速查/备忘图”组织，让用户只看图片也能读完主体内容。
- **不要在图片上写功能标签**：图上不要出现“封面、对标卡、对比卡、痛点卡、方法卡、金句卡、备忘卡、内页、图1、图2”等制作/结构标签；这些只能写在脚本、文件名或说明文档里。图片上的文字必须是用户会愿意收藏的正文内容。
- 有可用出图工具时，优先调用 FMGO `gpt-image-2` 或 `scripts/generate_images_gateway.py` 等同等流程完成实际生成；没有 key 或接口失败时，仍要把可补跑的配图脚本、提示词、清单和占位目录交付完整，并在结果中说明未生成的具体原因。

### 7. 交付组织
- 区分“工作区”和“最终交付区”：采集表、分析、提示词、manifest、补图清单、脚本等过程材料只能放在 `_internal` / `_work` / `00_config` / `01_raw_notes` / `02_analysis` / `03_remix` 这类工作区，不能混入最终发布包。
- 当用户要“小红书图文发布包/整包交付/直接发”时，最终交付区每篇只保留两类东西：
  1. `图片/` 或只包含图片文件的 `图片内容.zip`；
  2. `发布包.txt`。
- `发布包.txt` 只包含可选标题、正文、话题。不要把执行说明、对标分析、提示词、生成状态、manifest、补图清单、图片脚本写进去。
- 图片链路按“封面强钩子 → 痛点/判断 → 方法/框架 → 清单/步骤 → 总结/转发理由”组织。封面必须第一眼抓眼球，内页必须信息量足够、可阅读、可收藏，不能只有氛围或空泛情绪。
- 如果实际图片未生成，不能把该篇当作完整图片交付；待补图资料仍放入 `_internal` / `_work`，最终交付里只留下明确的缺图占位或等待补图状态。
- 交付字段和验收清单见 `references/delivery-workflow.md`。

## 排障模式工作流

只检查与当前问题直接相关的部分：

### 登录 / cookies / 风控
- 需要真实采集时，再执行 `xhs status --json` 或读取原始 API 响应。
- 区分是 `not_authenticated`、账号风控、空数据还是工具安装问题。
- 不要只看“已安装”，要看真实响应。
- 详细 SOP 见 `references/runbook.md`。

### Agent-Reach
- Agent-Reach 是“多平台聚合/胶水层”，不是小红书限制突破器。它可以统一调度网页、Twitter、Reddit、B站、小红书等渠道，但小红书渠道底层仍依赖 `xhs-cli` / `xiaohongshu-cli` 或小红书 MCP 的登录态。
- 检查顺序：`where agent-reach` / `agent-reach doctor` → `agent-reach install --channels=xiaohongshu` → `xhs status --json` → 真实搜索命令。不要只因为 Agent-Reach 已安装就判断小红书可采。
- 用户给出的 `npm install Agent-Reach --repository ...` 不是有效 npm 安装路线；Agent-Reach 当前应按 Python CLI / GitHub 项目处理。若需安装，先用 `agent-reach install --channels=xiaohongshu --dry-run` 或 `--safe` 看清会做什么。
- `agent-reach configure xhs-cookies` 当前更偏向写入 `xiaohongshu-mcp` Docker 容器/本地 MCP cookie 文件；它不等同于刷新 `xiaohongshu-cli` 的 `~/.xiaohongshu-cli/cookies.json`。如果 `xhs status --json` 返回 `Session expired`，仍需重新登录/扫码或重新导出有效 cookie。
- 如果用户问“能不能突破限制”，明确回答：不能承诺突破登录、验证码、风控或平台访问控制；只能在用户有权限、登录态有效、接口未风控的前提下读取。限制出现时走重新登录、换正常账号、降低频率或公开资料兜底。
- 具体安装、诊断、cookie 路径与踩坑见 `references/runbook.md`。

### GPTimage2 / FMGO 出图
- 用户已指定 FMGO 为小红书正文型配图的优先出图路线：`https://www.fmgo.top/v1`，模型 `gpt-image-2`。执行时从安全来源读取 API Key，例如临时环境变量 `FMGO_API_KEY`，不要把密钥写进技能、交付包、日志或回复。
- 调用方式原则：请求 `/images/generations` 或兼容端点，`prompt` 里写清楚图片上的完整中文标题、正文要点、层级排版、视觉风格、禁用词；让模型直接生成带文字的成品图。
- 严禁用 Python/PIL/canvas/HTML 截图把文字画到图片上来替代生图。Python 只能做接口调用、保存返回图片、重试失败项、生成 manifest、校验文件大小和数量。
- 接口不可用时，至少交付每篇的多图内容脚本、配图提示词、占位目录和补图清单，不能只写一句“接口不可用”。
- 已生成图片时要落盘，并视需要更新发布表。

## 输出原则

### 咨询模式
输出分析结论、结构拆解、改写结果、提示词、轻量建议表。

### 执行模式
按用户需要输出以下一种或多种：
- 原始采集表
- 爆款结构分析
- 二创文案表
- 图片识别分析
- 每篇必备配图成果：最终实际图片；如果未生成，待补图脚本、提示词、补图清单只能放入 `_internal` / `_work`，不得混入最终交付包
- 最终发布包：每篇只保留 `图片/` 或只含图片的 `图片内容.zip`，以及只含可选标题、正文、话题的 `发布包.txt`
- 已生成的二创图片
- 发布素材文件夹
- 发布总表

### 排障模式
输出：
- 问题定位
- 证据
- 原因判断
- 下一步修复动作
- 是否可切到兜底模式继续交付

## 质量检查

交付前确认：
- 来源与对标原因可追溯。
- 文案不是逐句洗稿，而是结构借鉴后的重写。
- 图片二创保留爆款抓眼球点，但替换侵权/可识别/照搬部分。
- 真实抓取失败时，没有伪造互动数据。
- 没有把 API Key、cookie 值写进交付文件或日志。
- 没有真实参考图时，不冒充 image2image。
- 每篇小红书二创/发布包最终只保留两类文件：`图片/` 或只含图片的 `图片内容.zip`，以及只含可选标题、正文、话题的 `发布包.txt`。
- 中间文件不得混入最终交付包：`图片内容脚本.md`、`配图提示词.json`、manifest、补图清单、采集表、分析文档只能放入 `_internal` / `_work` 工作区。
- 没有因真实采集失败、API Key 缺失、接口超时而省略配图环节；只能降级为工作区里的“提示词/脚本/待生成清单”，并在最终交付里明确图片未完成状态。
- 交付前检查最终图片文字：不得出现“封面、对标卡、对比卡、痛点卡、方法卡、金句卡、备忘卡、内页、图1、图2”等内部制作标签。

## 可直接复用的执行话术

### 咨询模式
> 我先不直接拉整套采集流程，先帮你拆这批对标内容的爆款结构、可借鉴点和二创方向。需要的话，我再继续扩成采集表、改写稿或图片二创方案。

### 执行模式
> 我会按“关键词筛选对标 → 结构分析 → 文案二创 → 图片二创 → 交付整理”的流程跑，但先从轻量版开始：先抓少量样本做判断，确认方向没问题后再扩到完整批量交付，这样更快也不容易卡住。

### 排障模式
> 我先不重跑整条链路，先定位你卡在哪一段：是登录态、风控、采集工具、还是 GPTimage2 接口。确认故障点后，再决定修链路还是直接走兜底交付。
