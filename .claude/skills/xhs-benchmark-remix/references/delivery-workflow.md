# 小红书对标采集与二创执行与交付规范

这个文件承载重型执行细节。只有在用户明确要进入执行模式、批量采集、批量二创、完整交付或排查图片落盘时再读取，不要默认在每次触发时全部展开。

## 1. GPTimage2 配置

用户指定用 **GPTimage2** 做图片二创，走 OpenAI 兼容图片接口：

```json
{
  "base_url": "https://www.geeknow.top",
  "api_key_env": "GEEKNOW_API_KEY",
  "model": "gpt-image-2"
}
```

执行时不要把 API Key 明文写进交付文件或日志；应优先放进环境变量 `GEEKNOW_API_KEY`。如果环境变量不存在，再让用户确认是否临时使用他提供的 key。

图片二创默认采用 **image2image**：先识别下载原图，再基于原图生成一组“保留核心抓眼球点但视觉上重新表达”的多图卡片。若没有真实参考图，则切到原创多图卡片生成/规划，不能省略配图。

## 2. 配图强制交付原则

用户已明确纠正：每一次小红书二创/图文/发布包任务里，“配图”是必然存在的，不是可选项。以后不能因为真实采集失败、没有图片 API Key、接口超时、任务是轻量版，就把配图环节省略。

### 2.1 最低交付标准

必须分清“最终交付”和“内部工作区”。用户最终要拿去发布的包，只保留两类东西：

```text
note_xxx_主题名/
├── 图片/                     # 或只含图片的 图片内容.zip
│   ├── 01.png
│   ├── 02.png
│   ├── 03.png
│   ├── 04.png
│   └── 05.png
└── 发布包.txt                 # 只含可选标题、正文、话题
```

`发布包.txt` 只放：

```text
可选标题：
1. ...
2. ...
3. ...

正文：
...

话题：
#... #...
```

中间文件不能进入最终交付目录：`图片内容脚本.md`、`配图提示词.json`、`manifest.json`、`补图清单.md`、采集表、分析文档、任务简报、执行日志都只能放在 `_internal` / `_work` / `00_config` / `01_raw_notes` / `02_analysis` / `03_remix` 等工作区。

要求：

- 有图片 API Key / 可用出图工具时，应生成实际 PNG/JPG，并校验文件落盘、数量、路径和发布表状态。
- 没有 API Key 或接口失败时，仍必须在内部工作区生成多图内容脚本、配图提示词、补图清单和占位目录；这叫“待生成配图包”，不是“无配图”，但这些中间文件不放进最终发布包。
- 真实小红书采集失败时，配图不能被删除；应改成“公开资料兜底 + 原创多图卡片方案/原创生成图”。
- 交付汇报里要明确区分：`已生成图片`、`待补图但提示词齐全`、`接口失败需重试`，不能含糊写“完成”让用户以为已经有图。

### 2.2 补图清单字段

如果任何一张图未生成，`补图清单.md` 或发布总表必须写清，但只能保存在 `_internal` / `_work` 工作区，不能进入最终发布包：

- note_id / 主题；
- 应生成图片数量；
- 已生成图片数量；
- 缺失图片编号；
- 未生成原因：缺少 API Key / 接口失败 / 超时 / 待人工确认 / 需参考图；
- 可直接复制的补跑命令或下一步动作；
- 对应 `配图提示词.json` 路径。

## 4. 搜索与筛选对标笔记

### 4.0 OneAPI 主路径与成本控制

当前小红书真实采集优先使用 OneAPI / GetOneAPI，除非用户指定其他工具或接口不可用。

**调用顺序：**

```text
大模型扩展关键词
↓
人工/模型筛 3-5 个最值得搜的词
↓
OneAPI 搜索：/api/xiaohongshu/search_note
↓
本地缓存与去重
↓
按收藏、评论、分享、综合分筛 Top 笔记
↓
OneAPI 详情：/api/xiaohongshu/fetch_video_detail
↓
必要时抓少量评论：/api/xiaohongshu/fetch_video_comment
```

**缓存规则：**

```text
cache_key = keyword + sort + noteType + page + date
```

24 小时内相同搜索优先复用缓存，不重复付费。交付表里标记 `from_cache=true/false`。

**默认成本预算：**

- 普通选题：搜索 3 次、详情 5 条、评论 1-3 条。
- 重点选题：搜索 5 次、详情 10 条、评论 3-5 条。
- 大规模调研：不要用 OneAPI 无脑逐条搜，改用批量服务或先确认预算。

**筛选指标：**

- `互动总分 = 收藏数 × 1.5 + 点赞数 × 1.0 + 评论数 × 2.0 + 分享数 × 2.0`
- `收藏/点赞比`：判断干货收藏价值。
- `评论/点赞比`：判断争议、共鸣、咨询需求。
- `分享/点赞比`：判断转发传播价值。
- `低赞高藏`：小而美干货样本，适合做清单/方法论。
- `高评低赞`：情绪/争议样本，适合提炼痛点和评论区语言。



- 搜索排序优先级：最多收藏/最热 > 最多点赞 > 综合。
- 类型过滤：默认只要图文笔记；如果用户要求视频，再扩展视频。
- 每篇记录字段：关键词、标题、正文、作者、链接、发布时间、点赞数、收藏数、评论数、分享数、图片链接/本地路径、标签、首图描述、备注。
- 评分公式默认：`爆款分 = 收藏数 × 1.5 + 点赞数 × 1.0 + 评论数 × 2.0 + 分享数 × 2.0`。如果收藏缺失，则用点赞和评论排序。
- 去重：同一链接、同一标题高度相似、同一作者连续多篇相似内容都要降权。

筛选输出：

- 每个关键词保留 Top N。
- 同时保留 3-5 篇“结构特别好但数据不是最高”的灵感样本。
- 标注“为什么值得对标”：选题、标题、封面、痛点、清单结构、评论区需求等。

## 5. 热点编排

热点编排不是硬蹭热搜，而是把“当日讨论情绪/热词/事件”转成更容易被点击、收藏和转发的内容入口。只有当热点与赛道自然相关，或用户明确要求“结合热点/当日热点/蹭热点/热点编排”时才启用。

### 5.1 热点来源

优先收集与关键词同赛道或相邻赛道的热点：

- 小红书站内搜索联想词、热搜词、相关话题、评论高频问题。
- 抖音/微博/知乎/百度热点中与目标人群有关的事件、情绪、争议或新词。
- 商业 API / 舆情工具返回的热词、话题趋势、热门笔记、评论情绪。
- 用户提供的行业事件、节日节点、政策变化、品牌活动或爆款案例。

如果不能真实抓取热点，不要伪造“全网热度”或“站内高赞”；只能标注为“公开资料/人工观察/待实测”。

### 5.2 热点相关性判断

每个热点先做筛选，不合适就放弃：

| 维度 | 判断 |
|---|---|
| 人群相关 | 目标用户会不会关心？ |
| 痛点相关 | 能不能引出真实问题，而不是只借一个词？ |
| 干货承接 | 能不能落到方法、清单、步骤、避坑、判断标准？ |
| 风险可控 | 是否涉及灾难、隐私、政治、医疗金融承诺、攻击个人？ |
| 时效价值 | 今天结合是否更容易点击，过几天是否还可复用？ |

默认只采用“强相关/中相关”的热点。弱相关热点不要硬蹭，宁可不用。

### 5.3 热点编排公式

把热点变成可收藏内容时，按这个顺序组织：

```text
当日热点/讨论情绪 → 目标人群的真实困惑 → 可迁移判断框架 → 可执行清单/步骤 → 评论区互动/转发理由
```

常用结构：

1. **热点切口**：用一句话说明今天大家为什么突然关注这个问题。
2. **人群转译**：把热点翻译成目标用户自己的处境，例如“这其实是在提醒普通人……”
3. **方法论承接**：给 3-5 条判断标准、操作步骤、避坑清单或复盘问题。
4. **收藏设计**：把中间图做成清单、表格、流程、对照题，让用户有保存理由。
5. **转发设计**：结尾给一句能转发给朋友/同事/伴侣的提醒，而不是只喊口号。

### 5.4 输出字段

启用热点编排时，在分析表或发布表里增加：

| 字段 | 说明 |
|---|---|
| hot_topic | 热点/热词/事件名称 |
| hot_source | 来源：站内搜索/公开热点/API/用户提供 |
| hot_relevance | 强相关/中相关/弱相关/放弃 |
| hot_angle | 借势角度：情绪、争议、方法、避坑、清单等 |
| evergreen_value | 去掉热点后是否仍有收藏价值 |
| risk_note | 版权、肖像、品牌、敏感议题、夸大承诺风险 |

### 5.5 质量要求

- 热点只是入口，主体必须是可收藏的判断、方法、步骤、清单或避坑原则。
- 不要把热点人物、品牌、争议细节做成侵权或攻击性内容。
- 不要为了蹭热度牺牲账号定位；热点要服务选题，不要抢走选题。
- 如果热点只能带来点击，不能带来停留、收藏或转发，就不要使用。

## 6. 下载与归档素材

- 下载原图到 `01_raw_notes/images_original/关键词/笔记ID/`。
- 文件名使用序号和来源 ID，避免中文过长导致路径问题。
- 保存 `source.md`，包括原链接、作者、采集时间、互动数据、原文摘要。
- 下载失败时保留图片 URL，不要静默丢失。

## 7. 爆款结构分析

从对标样本中提炼，而不是简单复制：

- 标题钩子：痛点型、结果型、清单型、反常识型、避坑型、低门槛型。
- 首图策略：大字报、前后对比、清单拼图、强结果展示、情绪表情、产品/场景中心。
- 正文结构：开头共鸣、问题展开、步骤/清单、证据、行动引导。
- 评论触发点：争议、求资料、求链接、求清单、晒结果。
- 可迁移元素：结构、角度、选题、排版逻辑。
- 不可照搬元素：原图、独特经历、品牌素材、作者口头禅、可识别人物/住址/隐私。

## 8. 文字二次创作

文字二创的目标不是完全另起炉灶，而是“保留爆款语气、内容核心和抓人节奏，同时换掉表达细节”，让它像同赛道新笔记，而不是原文搬运。

对每条拟发布内容至少产出：

- 3 个标题候选：强钩子、关键词友好、不过度标题党。
- 1 篇正文：保留爆款结构，但换人群、场景、细节、表达和观点。
- 8-15 个标签：赛道词、场景词、人群词、问题词、品牌/产品词（如适用）。
- 1 个评论区引导：例如“要清单扣 1”“你最卡在哪一步？”

改写原则：

- 先识别原文的语气：吐槽、种草、姐妹口吻、避坑、清单、故事感、测评感等。
- 保留核心内容：痛点、利益点、冲突点、结果感、评论诱因。
- 保留节奏：开头抓人、中段给料、结尾引导互动。
- 改掉具体表达：句子、例子、细节、顺序、个人经历、品牌露出都要重新组织。
- 不逐句同义替换；先抽象成“内容骨架”，再重新写。
- 加入用户账号的人设、经验、案例或观点。
- 避免绝对化承诺：如“必瘦、稳赚、根治、100%有效”。
- 如果是商业内容，明确卖点和转化路径，但不要牺牲平台语感。

## 9. 图片识别与 GPTimage2 图生图二创

图片二创默认用 **GPTimage2 image2image**，但这里的重点不是“有空再做图”，而是：小红书图文交付必须有配图成果。能出图就出实际图片；不能出图就交付可执行的多图脚本、提示词、补图清单和占位目录。流程分三步：

### 9.1 先识别下载图

对每张下载图先做视觉拆解，写入 `02_analysis/image_understanding.md`：

- 这张图最抓眼球的 1-3 个点是什么？例如大字标题、强对比、人物表情、结果展示、清单密度、颜色冲突。
- 它在整篇笔记里的作用是什么？封面吸点击、过程解释、结果证明、清单保存、情绪共鸣。
- 画面结构是什么？主体、背景、文字区、留白、色彩、构图、镜头距离。
- 哪些元素可以保留为“创意核心”？
- 哪些元素必须替换，避免照搬？人物脸、商标、住址、原作者独家照片、原版排版、完整文字截图等。

### 9.2 再写 image2image 提示词

把识别结果转成 `03_remix/image2image_prompts.md`。提示词遵循：

- 保留核心抓眼球点：例如“醒目的大标题 + 强结果对比 + 高收藏清单感”。
- 简单二创，不要过度改到失去原爆款味道。
- 替换视觉表达：换背景、换构图、换色彩、换人物/道具、换字体排版、换局部文字。
- 保留小红书图文语感：清晰、强钩子、适合手机竖屏、信息密度高但不乱。
- 输出比例默认 3:4 或 4:5；如原图是封面，优先 3:4。

示例提示词结构：

```text
基于参考图做 image2image 二创。保留它最抓眼球的核心：{核心抓眼球点}；保留类似的情绪和小红书爆款氛围，但不要照搬原图。
重新设计画面：{新的主体/场景/背景/配色/版式}。
文字区域：保留“大标题强钩子 + 3-5 个关键信息点”的结构，但替换成新的表达：{新文案}。
要求：手机竖版 3:4，清晰干净，高点击感，高收藏感，适合小红书图文封面/内页；避免出现原作者水印、商标、真实人物可识别脸、原图完整排版。
```

### 9.3 调用 GPTimage2 生成新图

- 使用 OpenAI 兼容接口：`base_url=https://www.geeknow.top`，模型 `gpt-image-2`。
- 优先用环境变量 `GEEKNOW_API_KEY` 读取 key；也兼容 `OPENAI_API_KEY`。不要把 key 写进脚本、Markdown、CSV 或日志。
- 每张图至少生成 1 张二创图；封面图可生成 2-3 个候选。
- 生成结果保存到 `03_remix/images_remixed/笔记ID/`，最终选用图复制到 `04_publish_pack/note_xxx/images/`。
- 如果没有真实参考图，必须切到原创多图卡片模式：生成或规划“可直接发布的原创小红书多图卡片”，不要声称已做 image2image。
- 如果接口暂时不可用，仍要输出完整的 image2image/原创配图提示词、图片内容脚本、补图清单和占位目录，方便稍后补跑。

可直接复用脚本：`scripts/generate_images_gateway.py`。它会读取每个 `04_publish_pack/note_xxx/image_prompt.txt`，调用 `/v1/images/generations`，失败时可自动尝试 `/v1/chat/completions` 包装，并写入 `03_remix/image_generation_results.json`。

推荐命令：

```bash
GEEKNOW_API_KEY="只放环境变量，不写入文件" PYTHONIOENCODING=utf-8 python "<skill_path>/scripts/generate_images_gateway.py" \
  --root "D:/yiren/抓取并转化/outputs/任务目录" \
  --base-url "https://www.geeknow.top" \
  --model "gpt-image-2" \
  --size "1024x1536" \
  --original \
  --update-sheet
```

参数：

- `--original`：无参考图时原创生成；有真实原图时不要加这个参数，而应使用 image2image 能力或输出补跑提示词。
- `--update-sheet`：同步更新 `04_publish_pack/publish_sheet.csv` 的 `image_generation_status`、`image_folder`、`publish_status`。
- `--limit N`：先试跑前 N 条，确认接口可用后再全量生成。

如果 API Key 已经出现在对话或 shell 命令历史中，提醒用户后续轮换 key。

每条内容输出 `image_edit_plan.md`：说明每张图的作用、识别结果、二创保留点、替换点、GPTimage2 提示词和生成状态。

## 10. 最终交付

交付至少包含：

1. `publish_sheet.csv/xlsx`：一行一条待发布笔记，字段包括标题、正文、标签、图片文件夹、来源链接、风险备注、建议发布时间。
2. `04_publish_pack/note_xxx/`：每条笔记独立文件夹。
3. `benchmark_analysis.md`：对标样本分析和二创策略。
4. 每篇必须包含配图成果：实际 PNG/JPG；或在无法出图时包含 `图片内容脚本.md`、`配图提示词.json`、`配图/` 占位目录、`补图清单.md`。
5. 如果 GPTimage2 已生成二创图，放入对应 `images/` / `配图/`；如果接口不可用，不能只交付一句失败原因，必须交付可补跑的提示词和补图清单。

### 10.1 按篇独立发布文件夹规范

当用户要图文发布包、批量二创、可直接发布素材时，不要只做总表。每一篇小红书内容都要有独立文件夹，方便用户直接复制发布。

建议结构：

```text
04_publish_pack/
├── note_001_主题名/
│   ├── 正文_可直接复制.txt
│   ├── 话题.txt
│   ├── 图片内容脚本.md
│   ├── 配图提示词.json
│   ├── 补图清单.md
│   └── 配图/
│       ├── 01.png
│       ├── 02.png
│       ├── 03.png
│       ├── 04.png
│       └── 05.png
└── note_002_主题名/
```

要求：

- `正文_可直接复制.txt` 只放发布正文，不放执行说明、对标分析、提示词、生成状态。
- `话题.txt` 单独放井号话题，便于复制。
- `图片内容脚本.md` 记录每张图要表达的内容，方便人工检查。
- `配图提示词.json` 记录每张图提示词，方便补跑或重做。
- `配图/` 放最终 PNG/JPG，文件名按顺序命名。

### 10.2 多图数量与风格一致性

小红书高赞高藏的图文笔记通常不是单图。除非对标样本明确是单图，或用户只要求单张封面，否则按多图笔记交付。

原则：

- 对标样本有几张图，二创时尽量做几张图；没有明确参考数量时，每篇默认 4-6 张，推荐 5 张。
- 同一篇笔记内所有图片必须风格一致：同一色系、同一字体气质、同一边距、同一信息层级、同一视觉符号。
- 不同篇可以有轻微主题差异，但整个发布包最好保持同一账号视觉系统。
- 多图不是“封面 + 几张氛围配图”，而是完整阅读链路：用户只看图片，也能读完主体内容。

推荐 5 图结构：

1. 强钩子图：一句话击中痛点或结果。
2. 症状/误区/判断图：让用户觉得“说的是我”。
3. 方法/步骤/清单图：给可执行内容，增强收藏欲。
4. 行动/金句/总结图：形成情绪收束和转发理由。
5. 速查/备忘图：做成可截图保存的清单或表格。

### 10.3 图片文字内容规范

用户反馈过：图片不能只是氛围配图，小红书图文的主体阅读发生在图片里。因此图片上的文字要承担正文主体。

要做：

- 图片文字直接写用户会收藏的内容：清单、步骤、判断、提醒、短句、表格。
- 标题要像小红书用户会点开的标题，而不是内部制作说明。
- 每张图的信息密度要比普通封面更高，但保持手机端清晰可读。
- 用“收藏起来”“照着做”“先做这 3 件事”等表达增强保存动机。

不要做：

- 不要在图片上写“封面、对标卡、对比卡、痛点卡、方法卡、金句卡、备忘卡、内页、图1、图2”等功能标签。
- 不要把“这是一张封面图/这是方法卡/这是对标卡”这类制作语言放进画面。
- 不要只做风景、人物、茶杯、床边等氛围图而缺少可阅读内容。
- 不要把对标来源、采集说明、执行状态、生成提示词写到图片里。

正确理解：这些结构标签可以出现在 `图片内容脚本.md`、文件名、提示词记录里，方便生产管理；但最终图片上只能出现面向读者的内容。

## 12. 质量检查清单

交付前逐项检查：

- 是否按点赞/收藏/评论等指标排序，并说明筛选逻辑？
- 是否保留来源链接和采集时间？
- 是否完成去重和人工可读的对标理由？
- 文字是否不是逐句洗稿，而是结构借鉴后的原创表达？
- 是否先识别原图，并提炼出最核心的抓眼球点？
- 是否用 GPTimage2 image2image 做了简单二创，或至少留下可补跑的提示词？
- 图片二创是否保留爆款味道，但替换了可侵权/可识别/照搬的部分？
- 每条内容是否能独立发布：标题、正文、标签、图片/图片方案齐全？
- 如果是公开资料兜底模式，是否明确标注“未直接抓取真实小红书高赞数据”，没有虚构互动数据？
- 是否验证 `xhs status --json` 或原始 API 响应，区分未登录、风控、无数据？
- 如果没有真实参考图，图片状态是否写成“原创生成/待 image2image 补跑”，而不是冒充图生图？
- 是否使用 `scripts/generate_images_gateway.py` 或同等流程把出图结果落盘，并更新 `image_generation_results.json` / `publish_sheet.csv`？
- 是否避免把 API Key、cookie 值写入任何交付文件、技能文件或日志？
- 如果用户要图文发布包，是否每篇都有独立文件夹，且正文、话题、配图分开存放？
- 每篇是否都有 `图片内容脚本.md`、`配图提示词.json`、`配图/` 目录；实际图片未全量生成时，是否有 `补图清单.md` 或发布总表里的明确待生成状态？
- 是否没有因为真实采集失败、API Key 缺失、接口超时而省略配图环节？只能降级为“提示词/脚本/待生成清单/占位目录”。
- 如果已经生成图片，是否逐篇校验图片数量、文件路径、落盘状态和发布表字段？
- 如果对标样本是多图或用户说“图文”，是否每篇生成或规划多张图，而不是只生成一张封面？
- 同一篇笔记内多张图是否风格一致，且形成完整阅读链路？
- 图片上的文字是否是面向读者的正文内容，而不是“封面/对标卡/对比卡/痛点卡/方法卡/金句卡/备忘卡/内页/图1/图2”等制作标签？
