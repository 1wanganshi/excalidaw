# 电商批量生图系统

## 目标
把“完整电商系统模块提示词 + OpenAI 兼容生图 API”变成一个可直接执行的批量生图工作流，而不是只产出静态提示词。

本技能的核心原则：
1. **绝不削减原始提示词**，原文是系统的总控规则。
2. **只做模块化拆分，不做语义阉割**。
3. **优先保产品一致性与商业用途**，其次才是风格增强。
4. **支持参考图输入**，优先走图生图。
5. **默认保存到 Windows 桌面**，输出必须可直接交付。

## 触发后你的工作流

### 第一步：收集最少必要输入
除非用户已经给全，否则只补最少信息：
- 产品名称或品类
- 核心卖点（1-3 个）
- 目标用户
- 价格带/定位
- 渠道用途
- 输出模式：精简版 / 标准版 / 完整版
- 参考图路径（可 1 张或多张）
- 是否立即执行生图，还是只先生成整套提示词

如果用户只说“按参考图批量出图”，你要默认：
- 输出模式 = 标准版
- 优先生成整套电商图组
- 优先走参考图图生图

### 第二步：读取规则，不要凭空改写
先读以下参考文件：
- `references/original_prompt_full.md`：原始完整提示词，必须视为最高规则来源
- `references/module_map.md`：原提示词模块拆解说明
- `references/api_workflow.md`：API 调用与批量执行流程

**要求：**
- 不要删减原提示词内容
- 不要把原提示词压缩成几个空泛段落
- 要以“模块调用”的方式复用原提示词逻辑

### 第三步：产出结构化方案
先输出一份执行方案，结构固定如下：

# 电商生图方案
## 1. 需求简析
## 2. AI视觉方向建议
## 3. 推荐图种规划
## 4. 生成任务清单
## 5. 执行方式

其中“生成任务清单”必须列出：
- 图组名称
- 图名称
- 用途
- 横版/竖版
- 对应卖点
- 是否需要文案留白
- 是否依赖参考图
- 输出文件名建议

### 第四步：装配高密度 Prompt
你不能只写一条总 prompt。你必须为**每一张图**装配一条可执行长 prompt。

装配逻辑：
1. 从 `references/original_prompt_full.md` 保留原始总规则
2. 从 `references/module_map.md` 选择对应模块
3. 结合用户产品信息与图种任务
4. 产出单图 prompt，至少包含：
   - 参考图一致性约束
   - 商业用途
   - 主体定义与角度
   - 构图与留白
   - 场景 / 背景 / 材质
   - 光影与氛围
   - 卖点视觉化表达
   - 反向限制词

补充规则：
- 标准版默认按约 20 张进行任务规划，而不是停在 8-9 张
- 当用户要求“拓展”“更丰富”“扩到 20 张”时，优先增加卖点图、场景图、竖版详情图、细节图、补充图
- 场景图默认允许加入人物手部、手持、半身生活方式人物，以增强真实代入感与种草效率
- 人物只能作为辅助叙事元素，不能压过产品主体，不能遮挡标签识别区，不能让画面变成人像大片
- 护肤 / 个护类默认优先扩展这类场景：人物手持展示、梳妆台使用、洗手台晨间护理、上脸前准备、半身护肤生活方式场景

人物场景图的卖点绑定规则：
- 每一张人物/生活方式场景图，都要先判断“这个场景要证明哪个卖点”，不能只写一个好看的生活场景
- 同一产品的不同场景图要承担不同卖点任务，避免所有场景都只写“高级、真实、好看”
- 按电商详情页做法处理：场景 = 使用情境，人物动作 = 使用证据，环境道具 = 卖点辅助，光影质感 = 品牌调性，留白区 = 后续文案承载
- Prompt 中必须写清：当前场景对应卖点、人物动作如何证明卖点、产品在画面中的识别位置、文案留白位置、禁止人物抢主体
- 护肤/个护示例：
  - 人物手持近景：主打“真实使用感 / 高颜值随手分享 / 包装精致”
  - 梳妆台晨间使用：主打“日常坚持使用 / 清爽不黏 / 晨间护理”
  - 洗手台晨间护理：主打“清爽肤感 / 快速融入流程 / 干净水润”
  - 上脸前准备：主打“补水修护 / 上妆前打底 / 肌肤稳定”
  - 半身生活方式人物：主打“都市女性生活方式 / 中高端信任感 / 种草代入”

画幅比例自动判断规则：
- 不要默认所有图都是 1:1；先判断渠道、图种、信息承载量，再决定 `size` 和 `orientation`
- 主图、商品卡、细节特写、结构展示：优先 `square`，常用 `1024x1024`
- 卖点说明图、人物场景图、竖版详情页、小红书种草图：优先 `portrait`，常用 `1024x1536`
- 横版详情首屏、Banner、活动横幅、PC 店铺头图：优先 `landscape`，常用 `1536x1024`
- 如果图中需要人物动作、使用流程、上脸前准备、场景叙事，通常不要压成 1:1，优先竖版以保留人物、产品和文案空间
- 如果图中需要右侧/左侧大面积标题文案区，优先横版
- 每个 job 的 prompt、`orientation`、`size` 必须一致，不要出现“竖版构图但 size 写 1024x1024”的冲突

### 第五步：需要落地生图时，写配置并执行脚本
当用户要求“直接生成图片”“批量出图”“落到桌面”“开始跑图”时，必须执行以下动作：

1. 在技能目录外创建一次性工作目录，建议：
   - `%USERPROFILE%\\Desktop\\电商批量生图输出\\<时间戳>\\`
2. 写入任务配置 JSON
3. 调用 `scripts/generate_ecommerce_images.py`
4. 将生成结果保存到桌面目录
5. 输出结果清单

## 输出目录规范
默认输出到：
- `C:\\Users\\<用户名>\\Desktop\\电商批量生图输出\\<时间戳>\\`

标准版建议任务结构：
- 主图组：5 张
- 卖点图组：3 张
- 场景图组：4 张
- 横版详情头图组：2 张
- 竖版详情图组：3 张
- 细节 / 材质图组：2 张
- 补充图组：1 张
- 总数：约 20 张

如用户明确要“更完整”或“全部都要”，可继续扩到 20+。

## 执行脚本调用方式
先确认 Python 可用，再运行：

```bash
python "<skill_dir>\\scripts\\generate_ecommerce_images.py" --task "<task_json_path>"
```

在 Windows 环境下优先使用绝对路径。

## 任务 JSON 约定
任务 JSON 至少包含：
- `api.base_url`
- `api.api_key`
- `api.model`
- `output_dir`
- `reference_images`
- `product`
- `mode`
- `jobs`

每个 `job` 至少包含：
- `name`
- `file_name`
- `prompt`
- `size`
- `background`
- `quality`
- `orientation`
- `use_reference`

## API 默认值
如果用户没有重新指定，默认采用：
- `base_url`: `https://www.geeknow.top/v1`
- `model`: `gpt-image-2`

**安全要求：**
- 不要把用户 API Key 硬编码进技能文件
- API Key 应写入任务 JSON 或环境变量，仅用于本次执行
- 在展示配置时对 key 做脱敏，例如只显示前 6 位和后 4 位

## 参考图策略
当用户提供参考图时：
1. 默认把所有参考图路径写入 `reference_images`
2. 若接口支持编辑类端点，优先走 `/images/edits`
3. 若编辑失败，则记录错误，并提示：
   - 接口可能不支持参考图编辑
   - 可退回文生图或更换兼容端点
4. 如果参考图不足以覆盖多角度，必须提醒用户补更多图，但仍继续执行

## API 返回校验与图片目录纯净要求
- `images/` 目录只能保存真实图片文件，允许扩展名通常为 `.png`、`.jpg`、`.jpeg`、`.webp`。
- 如果接口返回 `text/html`、接口首页、登录页、错误页或任何非图片内容，即使 HTTP 状态码是 200，也必须视为失败，绝不能保存到 `images/` 目录，更不能把 `.html` 统计为成功图片。
- 判断成功不能只看脚本退出码或接口 `status_code=200`，还要确认：
  - JSON 响应中存在可下载图片字段，如 `data[].url`、`data[].b64_json`、`b64_json`；或
  - 二进制响应的 `Content-Type` 必须以 `image/` 开头。
- 每轮执行后必须清点 `images/` 目录：真实图片数量、HTML/非图片误文件数量、失败任务数量。
- 若发现 `.html`、`.txt`、错误页等误文件，立即从 `images/` 删除，并把对应任务标记为失败或重跑，不要让用户误以为已经生成成功。
- 对于 OpenAI 兼容中转站，如果用户给的是根地址如 `https://example.com/`，优先规范化为 `https://example.com/v1` 再调用 `/images/generations` 或 `/images/edits`，避免请求到网页前端。
- 如果 `/images/edits` 返回非图片或 HTML，不要保存；可记录错误后尝试 `/images/generations` 兜底，但最终仍必须按真实图片文件数量汇报。

## 执行时必须做的文件落地
在真正调用脚本前，你必须：
1. 把每张图的最终 prompt 单独保存为 `.txt`
2. 把完整任务写成 `task.json`
3. 运行后生成 `run_log.json`
4. 最终生成 `summary.md`

## 给用户的最终反馈格式
执行完成后，用这个结构回复：

### 执行完成
- 输出目录：`...`
- 成功生成：X 张
- 失败：Y 张
- 使用模型：`gpt-image-2`
- 是否使用参考图：是 / 否

### 文件说明
- `prompts/`：每张图对应的最终长 prompt
- `images/`：生成图片
- `task.json`：本次任务配置
- `run_log.json`：接口调用结果与错误记录
- `summary.md`：本次图组说明

### 建议下一步
- 优先复看哪几张图
- 哪些图适合继续放大生成
- 是否建议补更多参考图

## 失败处理
如果脚本失败：
1. 先说明失败位置
2. 输出接口返回信息
3. 给出最小修复建议
4. 不要谎称已经生成成功

## 不要做的事
- 不要删减原始总提示词
- 不要把所有图合并成一个泛泛 prompt
- 不要把图片保存到技能目录里
- 不要默认把 API Key 写死到代码
- 不要在没有落地文件的情况下声称“已完成批量生图”
